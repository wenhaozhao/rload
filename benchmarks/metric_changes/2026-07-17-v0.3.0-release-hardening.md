# Metric change: v0.3.0 replay RSS and P99 release hardening

## Scope

- Version: v0.3.0 development
- Date: 2026-07-17
- Commits: `80035e3` (optimization), `8fff7e2` (validation report)
- Validation report: `benchmarks/VALIDATION_2026-07-17_0.3.0-dev.md`

## Metric contract

- Replay memory: positive RSS increment of at most 256 B/access-log entry and
  a multi-size RSS slope of at most 256 B/entry.
- Replay throughput: median loss of at most 10% versus the static-request mode.
- Three-way latency: development P99 median absolute error of at most 5% versus
  wrk across at least five alternating paired runs.
- Three-way throughput: no more than 3% regression versus the installed rload
  release.
- Environment: macOS arm64, Rust 1.96.1, wrk 4.2.0, installed rload 0.2.4.

## Proposed optimization

Reduce access-log replay memory without changing the replay hot path. The
expected mechanism was to stop storing empty header and body vectors in every
`ReplayRequest`; those fields are needed by JSONL and custom requests but not by
ordinary Nginx access-log records.

## Regression evidence

Initial replay measurements failed the memory gate:

| Entries | Median RSS increment | Bytes/entry | Median throughput loss | Raw results |
|---:|---:|---:|---:|---|
| 100,000 | 27.58 MiB | 289.2 | -0.41% | `benchmarks/results/replay-20260717T141532Z-evhjOd` |
| 500,000 | 134.66 MiB | 282.4 | -1.60% | `benchmarks/results/replay-20260717T141712Z-8PynEB` |

The initial cross-size RSS slope was 280.7 B/entry, 9.6% above the gate.

The first five-run three-way result also failed the P99 accuracy gate with a
10.169% development median absolute error. Other development errors remained
normal: RPS MAE was 0.824%, P90 MAE was 1.449%, and `read_bytes` per-request MAE
was 0.0076%. Raw results:
`benchmarks/results/threeway-20260717T141219Z`.

## Cause analysis

Replay RSS scaled linearly across 100,000 and 500,000 entries. Code inspection
showed that every access-log `ReplayRequest` retained two empty `Vec` values and
a body-presence field until request encoding, creating per-entry capacity that
access-log replay never uses. The measured reduction after removing that layout
cost supports this as the primary cause.

The P99 failure was not accompanied by throughput, P90, byte-accuracy, error,
or RSS degradation. A clean five-run repetition passed while the installed
release itself showed a 10.027% P99 error in that repetition. The evidence
supports machine/scheduler tail variance rather than a demonstrated development
hot-path regression. This remains an uncertainty assessment, not a claim that
local P99 variance has been eliminated.

## Options considered

| Option | Expected effect | Correctness risk | Cost | Reversibility |
|---|---|---|---|---|
| Stream access-log parsing directly into encoded requests | Remove the intermediate representation | High: duplicates preparation paths and couples parsing to target encoding | High | Medium |
| Store request payload fields only when present | Save about 40-48 B/access-log entry | Low: accessors preserve existing validation and encoding | Low | High |
| Raise the 256 B/entry gate | No memory reduction | High: hides a measured regression | Low | High |

## Selected optimization or compensation

`ReplayRequest` now stores headers, body, and explicit-body state in an optional
boxed payload. Access-log records allocate no payload; JSONL and custom requests
retain the same header/body semantics. The metric gate was not changed.

For the P99 failure, no code compensation was applied because the evidence did
not isolate a development regression. The compensation was a clean, complete
five-run repetition with the same workload and paired ordering.

## Post-change validation

| Entries | Median RSS increment | Bytes/entry | Median throughput loss | Raw results |
|---:|---:|---:|---:|---|
| 100,000 | 24.34 MiB | 255.3 | -0.89% | `benchmarks/results/replay-20260717T142026Z-DjnUNl` |
| 500,000 | 116.66 MiB | 244.6 | -6.09% | `benchmarks/results/replay-20260717T142121Z-TOGppo` |

The post-change cross-size RSS slope was 242.1 B/entry. Both replay sizes,
their throughput results, and the scaling slope passed.

The clean three-way repetition passed with 0.497% development RPS MAE, 1.525%
P90 MAE, 3.476% P99 median absolute error, and 0.0128% `read_bytes` per-request
MAE. Mean development throughput was 0.03% below rload 0.2.4. Raw results:
`benchmarks/results/threeway-20260717T142325Z`.

## Guardrails and rollback

- Correctness checks: 43 library, one binary, 50 CLI, and 32 HTTP integration
  tests; Clippy with warnings denied; release build; package verification.
- Rollback: revert `80035e3` if payload semantics regress, accepting that replay
  RSS will return above the current gate until a replacement optimization lands.
- Revisit triggers: replay RSS above 256 B/entry at either validation size;
  slope above 256 B/entry; throughput loss above 10%; or two clean five-run P99
  failures above 5% on the same commit.
- Remaining follow-up: cross-platform CI and a separate-host P99 validation when
  that environment is available.
