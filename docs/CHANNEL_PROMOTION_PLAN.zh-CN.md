# rload 渠道推广任务计划

本计划把渠道推广纳入标准发布流程。它适用于 `v0.3.0` 及后续版本；它不授权自动发帖，也不替代渠道管理员或发布负责人的审批。

## 目标与边界

目标是让有真实流量回放需求的 Rust、SRE 和后端工程师，能在 60 秒内完成安装并跑通一个可复现实验，并把安装失败、场景缺口和版本问题沉淀为可追踪反馈。

首轮 30 天目标：50 个 GitHub Star、10 条真实 issue 或讨论、3 份外部使用反馈；不以浏览量或泛流量作为唯一成功标准。

不在渠道中承诺 HTTP/2、gRPC、Lua/LuaJIT、分布式执行或 GUI；不把特定环境的性能验证表述为通用性能保证。稳定版必须明确版本、验证范围和反馈入口。

## 发布包：所有渠道共用的事实源

发帖前一次性生成并由发布负责人确认，避免三处内容漂移：

- Release URL、标签和明确的「稳定版」定位；
- 定位：**用真实流量重放，而不是只压一个 URL**；
- 三个变化：YAML workload profile、CI 最终摘要断言、离线 HTML 报告；
- 已验证安装方式：`cargo install rload`，以及对应 Release/ Homebrew 安装链接；
- 可复制场景：`rload --requests 1000 --access-log ./access.log --replay-order shuffle --seed 42 https://staging.example.com/`；
- 反馈入口：GitHub Issues 或 Discussions；
- 验证依据：对应版本的 benchmark validation report；性能表述限定为该报告的测试环境。

命令中的 `staging.example.com` 必须保持为示例域名；不得鼓励读者对未获授权的目标执行压测。

## 分渠道任务卡

### QQ Rust 中文社群

**受众与目的**：面向熟悉 Rust 工具链的早期试用者，优先收集安装、跨平台二进制、wrk 迁移和日志格式兼容问题。

1. 取得群管理员允许，确认允许的发布形式、长度和链接规则；未经许可不群发或重复投放。
2. 发布 120–180 字短帖：问题场景 → rload 的日志/JSONL 重放能力 → 一条安装或回放命令 → 稳定版标识 → GitHub 反馈入口。
3. 首 48 小时由技术值班人跟进；对可复现问题请求最小命令、操作系统、rload 版本和脱敏日志样本，并转为 issue。

**验收**：管理员授权状态、帖子链接/截图、版本、负责人、讨论结果全部记录；每个有效问题都有 issue 或明确的无需处理结论。

### 知乎

**受众与目的**：用可复现实验解释「为什么单 URL 压测不足」，让读者理解真实流量回放、可重复性和 CI 断言的价值，而不是单纯展示功能清单。

1. 选择与压测、wrk、Nginx 日志、性能回归相关的高相关问题，或发布一篇技术文章；避免在不相关问题下做推广。
2. 文章结构固定为：真实场景/痛点 → 单 URL 的盲区 → Nginx access log 回放示例 → YAML profile 与 `--assert` 的 CI 用法 → 能力边界和反馈请求。
3. 给出完整、已验证命令和 GitHub Release 链接；性能数据只引用 validation report 的环境、方法与结论。
4. 发布后 72 小时集中答疑；将产品缺口、文档歧义与 bug 分类入库。

**验收**：内容链接、所答问题或文章主题、命令复验结果、评论中的有效反馈数、关联 issue 链接。

### 小红书

**受众与目的**：以工程实践的可读形式触达后端和 DevOps 从业者，把兴趣导向可验证的 GitHub 文档，而不是压缩技术细节到夸张结论。

1. 制作 5–7 页图文：封面「别再只压一个 URL」；真实流量回放的场景；日志到 staging 的流程；一条命令；YAML + CI 断言；稳定版反馈入口；能力边界与授权提醒。
2. 将小红书作为审核敏感渠道，至少提前一个工作日完成内容预审。正文、图片、封面、标签和评论回复均由内容负责人复核；审核未通过时使用已准备的简化版，不影响其他渠道节奏。
3. 正文使用短段落和工程实践语境，说明 rload 是 Rust HTTP/1.1 工具，支持 Nginx access log 与 JSONL 回放；明确稳定版版本号和验证范围。避免「最强」「唯一」「保证提升」「极限」等绝对或夸张表述，也不把实验数据包装成普适结论。
4. 不在封面或正文堆叠外链、联系方式、群号、二维码或引流措辞。GitHub Release、中文使用手册和反馈入口仅在平台允许的方式下提供；如链接受限，使用项目名称和可搜索的仓库名，并将技术反馈统一沉淀到 GitHub。
5. 不展示真实生产日志、域名、用户标识或未脱敏的监控截图；示例始终使用 `staging.example.com`、合成日志和授权压测提醒。
6. 发布后 72 小时回复安装与场景问题，将深入技术讨论引导至 GitHub，保留可追踪上下文。

**验收**：图文链接、审核状态、素材版本、允许的反馈入口、有效技术咨询数、被转化的 issue/Discussion 链接。若未过审，记录被拒素材与修改原因，不重复提交同一版本。

## 值班与反馈闭环

- 每个渠道必须有一名内容负责人和一名技术值班人；无人值班的渠道延后发布。
- 发布后首 48 小时的目标回复时间为一个工作日内；安全、误用压测或数据脱敏问题优先处理。
- 将反馈归类为：安装、兼容性、日志/JSONL 输入、性能理解、文档、功能请求、bug；不在社媒评论中承诺发布日期。
- 对每条渠道推广，在 `docs/release_promotions/<tag>.md` 留存标准记录，并增加渠道、链接、负责人、发布时间、有效反馈数量和关联 issue/Discussion。

## 发布阻断条件

以下任一条件未满足时，暂停外部推广而非临时修改文案掩盖问题：

- Release 不是公开状态，或稳定版标签与实际定位不一致；
- crate、二进制、Homebrew 或网站安装指引无法复验；
- 发布文档交接缺失且没有命名的后续负责人；
- 无法提供性能表述的验证依据；
- 未获得渠道管理员/账号发布授权，或没有发布后答疑能力。
