# Technical Design Proposal: Rload v0.3.0 (Declarative Config, CI Assertions, and Prometheus Export)

This document presents the detailed architectural and structural design for the features planned in **Phase 2 (v0.3.0)** of the Rload strategic roadmap. By establishing these designs beforehand, we ensure high maintainability, C-level performance parity, and developer-centric interfaces.

---

## 1. Declarative Test Profile Schema (`rload.yaml`)

To avoid verbose command-line executions in CI/CD environments, we introduce a version-controlled YAML schema. This file defines the full execution behavior, targets, constraints, and observability settings.

### 1.1 Draft Specification
```yaml
# rload.yaml
version: "v1"
metadata:
  name: "gateway-stress-test"
  description: "Staging gateway Nginx access-log high-volume replay"

target:
  url: "https://api.staging.internal"
  # Optional client TLS configuration for mutual TLS (mTLS) or custom CA roots
  tls:
    client_cert_path: "/etc/certs/client.crt"
    client_key_path: "/etc/certs/client.key"
    ca_cert_path: "/etc/certs/ca.crt"
    verify_server_cert: true

runner:
  threads: 4
  connections: 200
  duration: "10m"       # Supports "s" (seconds), "m" (minutes), "h" (hours)
  timeout: "2s"         # Request timeout duration
  co_correct: true      # Enable Coordinated Omission post-correction

load_profile:
  mode: log_replay      # "static" or "log_replay"
  
  # Configuration block for "static" mode (active if mode == static)
  static:
    method: "GET"
    path: "/v1/health"
    headers:
      Content-Type: "application/json"
      User-Agent: "rload-agent/0.3.0"
    body: ""

  # Configuration block for "log_replay" mode (active if mode == log_replay)
  log_replay:
    path: "/var/log/nginx/access.log"
    format: "nginx"      # "nginx" (standard combined format) or "jsonl"
    order: "sequential"  # "sequential", "shuffle", or "random"
    filter:
      whitelist_uris: ["^/api/v2/.*", "^/oauth/.*"]
      blacklist_methods: ["DELETE", "PUT"]
    pacing:
      mode: "stages"     # "none" (unlimited/as-fast-as-possible), "timestamp" (original pacing), "stages" (custom rates)
      stages:
        - duration: "1m"
          target_rps: 500  # Ramp-up stage
        - duration: "8m"
          target_rps: 5000 # Sustained load stage
        - duration: "1m"
          target_rps: 100  # Cool-down stage

observability:
  # Internal Prometheus scrape endpoint
  prometheus:
    enabled: true
    port: 9100           # Listens on http://0.0.0.0:9100/metrics
    path: "/metrics"

assertions:
  # Assertions are evaluated at the end of the test.
  # The process will exit with a non-zero exit code if any assertion fails.
  - expression: "p99 < 50ms"
    message: "Tail latency (P99) exceeded acceptable SLA"
  - expression: "error_rate < 0.005" # <= 0.5% errors (status >= 400 + socket errors)
    message: "Error rate exceeded 0.5% safety threshold"
  - expression: "status_errors == 0"
    message: "HTTP status errors occurred during replay"
```

## 2. Automated Gating & Assertion Engine (`--assert`)

We require a fast, low-overhead evaluation engine that parses basic boolean assertions against the final test metrics (`RunSummary`). To maintain the 3MB static memory footprint of `rload`, we will write a custom micro-lexer/parser in Rust without relying on heavy script interpreter crates (such as Lua, JavaScript, or general-purpose expression parsers).

### 2.1 Available Metrics in Assertion Context
* **`completed`**: Total requests completed successfully.
* **`socket_errors`**: Total count of connection, write, or read socket errors.
* **`status_errors`**: Total count of HTTP status codes `>= 400`.
* **`total_errors`**: `socket_errors + status_errors`.
* **`error_rate`**: `total_errors / (completed + socket_errors)`.
* **`mean`**: Average latency.
* **`p50`, `p75`, `p90`, `p95`, `p99`, `p999`**: Microsecond-precision percentiles computed by HdrHistogram.

### 2.2 Assertion Syntax & Unit Handling
The parser will interpret three components: `[METRIC] [OPERATOR] [VALUE]`.
* **Operators**: `<`, `<=`, `>`, `>=`, `==`, `!=`.
* **Value Units**:
  * Latency values (for `mean`, `p50`, `p99`, etc.) support unit suffixes: `us` (microseconds), `ms` (milliseconds), `s` (seconds). Suffixes are automatically parsed and normalized to microseconds for precise HdrHistogram comparison.
  * Ratio values (for `error_rate`) are parsed as floats (e.g. `0.01` represents 1%).
  * Counts are parsed as standard `u64` integers.

### 2.3 Evaluation Pipeline
1. At CLI startup, if `--assert` parameters or `assertions` array in `rload.yaml` are present, parse them into an AST represented as:
   ```rust
   pub struct Assertion {
       pub metric: MetricField,
       pub operator: Operator,
       pub target_value: Value,
       pub message: Option<String>,
   }
   ```
2. Run the load test to completion.
3. Fetch the aggregated `RunSummary` containing metrics and latencies.
4. Evaluate each parsed `Assertion` against `RunSummary`.
5. If any assertion evaluates to `false`, print details to `stderr` in standard CI error format:
   `::error file=rload.yaml,line=38::Assertion Failed: Tail latency (P99) exceeded acceptable SLA. Value: p99 = 84.2ms (threshold: < 50ms)`.
6. Terminate `rload` with exit code `1`.

## 3. Prometheus Scrape Server

To avoid dragging down performance or introducing socket locks, we implement a non-blocking scrape responder.

### 3.1 Architecture Design
The Prometheus scrape server will run on a dedicated thread or share the multi-plexed event loop of the main thread. To avoid lock contention with active worker threads that are pumping metrics, workers will write metrics to thread-local variables. An epoch-based atomic swap or a lightweight lock-free atomic aggregator will compile these statistics into a shared read-only snapshot every 250ms, which is then made available to the scrape responder.

### 3.2 Output Metric Schema
The responder will output standard Prometheus text format (`text/plain; version=0.0.4`):

