# rload v0.3.0-rc.1 frozen contract

This document freezes the additive profile, assertion, result, report, and
runtime-recovery contracts for the first v0.3.0 release candidate. Unknown
profile fields are rejected. Deferred fields are not accepted as no-ops.

## Profile v1

The top-level fields are `version`, `target`, `runner`, `load_profile`,
`observability`, and `assertions`. `version` must be `v1`.

- `target.url` is required and owns the complete request target, including the
  path used by static requests.
- `runner` supports `threads`, `connections`, `duration`, and `timeout`.
- `load_profile.mode` is `static` or `log_replay`.
- `load_profile.static` supports `method`, `headers`, and `body`.
- `load_profile.log_replay` requires `path` and `format`; `format` is `nginx`
  or `jsonl`. It also supports `order`, `seed`, `rounds`, `schema`,
  `skip_invalid_records`, `pacing`, and `filter`.
- replay `pacing.mode` is `none`, `rate`, `timestamp`, or `stages`. Rate uses
  `rate`; timestamp pacing uses optional `speed`; stages contain `duration` and
  `target_rps`.
- replay filters support `allowed_methods` and `allowed_uris`.
- `observability` supports `output_format` and `output_html`.
- each assertion requires `expression` and may provide `message`.

Explicit CLI values override profile values. Profile values override existing
CLI defaults. Cross-field validation completes before target connection.

TLS client certificates, custom CA fields, blacklist filters, Prometheus
configuration, and static `path` are not profile-v1 fields.

## Assertion grammar

Assertions use exactly `METRIC OPERATOR VALUE`, separated by whitespace.
Operators are `<`, `<=`, `==`, `!=`, `>=`, and `>`.

Supported metrics are:

- latency: `mean`, `p50`, `p90`, `p95`, and `p99`;
- rate: `rps`;
- counts and ratios: `error_rate`, `status_errors`, `socket_errors`,
  `completed`, `abandoned_requests`, and `recovery_attempts`.

Latency thresholds require `us`, `ms`, or `s`. Other metrics reject duration
units. CLI and profile assertions use the same evaluator. A failed assertion
returns a non-zero process status and writes its stable diagnostic to stderr;
profile messages prefix that diagnostic.

`p75`, `p999`, and `total_errors` are not assertion metrics in this contract.

## JSON schema v1 additions

The schema version remains `1`. Existing fields retain their names and meaning.
v0.3.0 adds:

- `summary.abandoned_requests` and `summary.recovery_attempts`;
- aggregate `latency.minimum_us`, `maximum_us`, `median_us`, and `p95_us`;
- per-method `minimum_latency_us`, `maximum_latency_us`, and
  `median_latency_us`.

`median_us` and `p50_us` use the same histogram calculation. Empty latency
statistics are `null`. The same input result produces byte-identical HTML.

## Runtime failure contract

Configuration, target parsing, replay loading, invalid HTTP responses, and TLS
certificate validation remain terminal. Runtime connection, TLS I/O, read,
write, and timeout failures are counted and isolated.

Duration runs recover until their deadline. Fixed-request runs have three
recovery attempts per logical request. Exhaustion increments
`abandoned_requests`; each actual recovery attempt increments
`recovery_attempts`. A retry after request bytes were written has at-least-once
semantics and can duplicate target-side effects.

The detailed reliability contract is
`designs/v0.3.0_failure_recovery.md`.

## Deferred

Prometheus, mTLS profile fields, custom CA configuration, HTTP/2, gRPC,
distributed execution, scripted hooks, and GUI/TUI work are outside v0.3.0.
