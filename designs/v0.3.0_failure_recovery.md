# v0.3.0 Runtime Failure Recovery Contract

## Reliability promise

For a valid workload, runtime transport failures do not turn the entire run
into a configuration failure. Duration-limited runs recover until their
deadline. Fixed-request runs use a finite per-request recovery budget; after
the budget is exhausted, the request is abandoned, counted, and the run exits
successfully with a partial-completion summary.

This is an **at-least-once** load-generation contract once any request bytes
have been written. `rload` cannot determine whether a failed peer processed a
partially transmitted request, so a retry can duplicate target-side effects.
Users must use idempotent targets, idempotency keys, or accept duplicates for
such workloads.

## Durable facts and intents

- The validated `RunConfig` and replay sequence define the requested workload.
- A completed HTTP response is the only successful completion fact.
- Socket-error counters record failed transport attempts by category.
- New `abandoned_requests` and `recovery_attempts` summary facts make fixed
  request partial completion and recovery cost observable.

## Boundary map

| Boundary | Failure modes | Required semantics | Recovery owner |
|---|---|---|---|
| Initial TCP/TLS connection | refusal, timeout, TLS failure | Retry only within the active run limit; fixed runs consume the current request budget | Worker connection |
| Request write | reset, write failure, partial write | Count write failure; retry may duplicate the request | Worker connection |
| Response read/parse | EOF, reset, timeout, malformed response | Count read/timeout failure; malformed HTTP remains terminal protocol failure | Worker connection |
| Replay input | unreadable/malformed input | Fail before network I/O | Configuration loader |
| Final summary | partial completion | Always expose completion, errors, recovery attempts, and abandoned requests | Run-summary merger |

## Idempotency and ordering

Retries preserve the current request; they do not fetch a later replay entry.
This preserves client-side replay order per connection but cannot guarantee
target-side ordering across concurrent connections. A retry after `write_offset
> 0` can duplicate execution. `rload` does not deduplicate responses because it
has no server-issued request identity.

## Failure classification

- **Retryable transport:** connect, TLS I/O, write, read, and timeout.
- **Terminal protocol:** malformed or invalid HTTP response.
- **Terminal preparation:** invalid options, target parsing, replay input, and
  profile validation.

## Recovery and reconciliation

The fixed-request budget is three recovery attempts per logical request. A
successful response resets the budget for the next request. On exhaustion,
`rload` stops that connection and increments `abandoned_requests`; it does not
invent a successful completion or loop forever. Duration-limited runs retain
their deadline as the recovery bound.

## Observability

Text, JSON, and HTML output must expose:

- categorized socket errors;
- `recovery_attempts`;
- `abandoned_requests`;
- requested versus completed work when the run is request-limited.

## Failure-injection plan

1. Permanently unavailable fixed target: exits within three recovery attempts
   per assigned request, reports partial completion and no process error.
2. Mid-request reset followed by healthy listener: retries and completes,
   with one read/write error and one recovery attempt.
3. Repeated timeout: stops at budget exhaustion for fixed runs; continues to
   the deadline for duration runs.
4. Malformed response: fails as a terminal protocol error without retrying.
5. Multi-connection mix: one abandoned connection does not prevent healthy
   connections from completing their assigned work.
